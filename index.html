STARBUKCS

1인 프로젝트

목표

실제 스타벅스 홈페이지 수준에 근접한 결과물을 목표로 해당 프로젝트를 기획하고 진행하였습니다.

주요 기능 및 특징

  • swpier를 이용한 슬라이더 구현
  • JavaScript를 이용한 동적 사이트
  • GSAP의 scrollTrigger를 기능을 사용한 반응형 웹 사이트
  • 사용자 환경에 따라 유연하게 반응할 수 있도록 rem 단위를 사용하여 다양한 화면 크기를 고려한 레이아웃을 구성하였습니다.

사용 기술/Tool

항목을 누르면 자세히 볼 수 있습니다.
➡️ vite

개발 생산성과 성능 향상을 위해 Vite를 활용하여 실시간 반영 기능을 적용하고, 최종 빌드 시 경량화 및 최적화를 수행하였습니다.

➡️ GSAP

GSAP ScrollTrigger 기능을 활용하여 특정 스크롤 구간에 맞춰 애니메이션을 트리거하도록 구성하였습니다.

➡️ Swiper

사용자 친화적인 슬라이더 인터페이스 구현을 위해 Swiper 라이브러리를 도입하여 효과적인 콘텐츠 전환을 구현하였습니다.

➡️ Lighthouse

Lighthouse 분석 결과: 성능(90점), 접근성(86점), 권장사항(78점), 검색엔진 최적화(91점)

SEO 및 웹 표준 준수 여부를 확인하고, 전반적인 웹사이트 품질 향상을 위해 Lighthouse를 활용하였습니다.

➡️ Web Developer

Web Developer의 HTML 유효성 검사 기능을 사용하여, 마크업 구조를 웹 표준에 맞게 구성하였습니다.

프로젝트 회고

➡️ Swpier

개인 프로젝트를 진행하던 중 캐러셀(Carousel) 기능이 필요한 부분이 있었고, 직접 구현하는 대신 더 효율적인 방법을 찾던 중
JavaScript 기반의 라이브러리인 Swiper를 알게 되어 도입하게 되었습니다.
Swiper를 통해 캐러셀 기능을 손쉽게 구현할 수 있었고, 이를 통해 개발 시간을 크게 단축할 수 있었습니다.

다만, 라이브러리를 사용할 때는 단순히 사용하는 것에 그치지 않고,
해당 라이브러리가 어떻게 동작하는지 구조를 이해해야만 원하는 방식으로 수정하거나 확장할 수 있다는 점도 함께 깨달았습니다.

이 경험을 통해 라이브러리의 편의성과 함께 구조에 대한 이해의 중요성을 깊이 느낄 수 있었습니다.

➡️ GSAP

개인 프로젝트를 진행하면서 여러 애니메이션이 필요한 부분이 있었고,
이를 모두 직접 구현하는 것은 비효율적이라 판단해 대안을 찾던 중 GSAP(GreenSock Animation Platform)
JavaScript 애니메이션 라이브러리를 발견하게 되었습니다.

GSAP는 복잡한 애니메이션을 간단한 코드로 구현할 수 있을 뿐만 아니라,
다양한 확장 플러그인을 통해 추가적인 기능도 손쉽게 적용할 수 있다는 점에서 큰 장점을 느꼈습니다.

무엇보다 필요한 기능만 선택적으로 플러그인으로 불러올 수 있는 구조는 성능 최적화에도 도움이 될 수 있겠다는 점에서 인상 깊었습니다.
이 경험을 통해 효율적인 개발과 최적화를 동시에 고려할 수 있는 라이브러리 선택의 중요성을 느낄 수 있었습니다.

목표

  • Git 및 GitHub를 활용한 체계적인 팀 프로젝트 관리
  • 불필요한 페이지 이동 없이 하나의 페이지에서 모든 기능 제공, 성능 향상에 기여

프로젝트 회고

➡️ 공용 함수 제작

팀 프로젝트를 진행하면서 역할을 나누어 작업했지만, 완료 후 확인해보니 유사한 기능의 함수들을 각자 따로 구현한 중복이 발생했습니다.
작업에 들어가기 전, 보다 구체적인 회의를 통해 공통 함수들을 먼저 설계하고 공유하고 공유했더라면,
중복 없이 더 효율적인 개발이 가능했을 것이라 생각했습니다.

프로젝트를 통해 작업 전 충분한 논의와 설계 과정을 거치는 것이 개발 효율성과 완성도에 있어 매우 중요하다는 점을 깨달았습니다.

➡️ GitHub Merge

팀 프로젝트 진행 중 GitHub에서 머지를 수행하던 중 충돌이 발생했고,
이를 충분히 확인하지 않은 채 병합을 진행해 제가 작업한 일부 내용이 유실되는 경험을 했습니다.

이 경험을 통해 Merge 충돌 시 변경 사항을 정확히 확인하고 신중하게 처리하는 것의 중요성을 깊이 느꼈습니다.

karly

팀 프로젝트

목표

컬리(Karly)의 실제 웹사이트를 클론하여 UI 및 기능을 구현한 프로젝트입니다.
HTML, CSS를 이용한 UI 구성에 집중한 프로젝트 입니다.

주요 기능 및 특징

  • 디바이스 해상도에 따라 UI가 동적으로 조정되어 일관된 사용자 경험을 제공하도록 설계하였습니다.
  • 모바일 환경에서도 원활하게 작동하도록 반응형으로 설계하였습니다.
  • 사용자 환경에 따라 유연하게 반응할 수 있도록 rem 단위를 사용하여 다양한 화면 크기를 고려한 레이아웃을 구성하였습니다.
  • UI 요소를 컴포넌트화하여 역할을 명확히 분담하고, 작업 충돌 없이 효율적인 협업을 구현하였습니다.

사용 기술

➡️ vite

개발 생산성과 성능 향상을 위해 Vite를 활용하여 실시간 반영 기능을 적용하고, 최종 빌드 시 경량화 및 최적화를 수행하였습니다.

➡️ Semantic HTML elements

웹 접근성과 유지보수성을 고려하여 HTML5의 시맨틱 태그를 적극 활용하였습니다.

➡️ Tailwind CSS

디자인 시스템의 일관성과 유지보수 효율성을 높이기 위해 Tailwind CSS를 적용하였습니다.

➡️ Lighthouse

SEO 및 웹 표준 준수 여부를 확인하고, 전반적인 웹사이트 품질 향상을 위해 Lighthouse를 활용하였습니다.

➡️ Git Commit

팀 프로젝트를 진행하면서 커밋 메시지에 한글과 영어가 혼용되어 사용되어 통일성이 부족했고,
이로 인해 각 커밋의 내용을 명확히 파악하기 어려운 점이 있었습니다.

커밋 메시지 작성 규칙 및 통일성의 중요성을 느끼는 계기가 되었습니다.

➡️ 공용 color CSS 작성

프로젝트 진행 시 공통으로 사용하는 색상을 일일이 찾아 직접 입력하는 비효율적인 작업이 있었습니다.

이를 개선하기 위해 공통 색상을 :root에 CSS 변수로 선언한 별도의 CSS 파일을 생성하고,
이를 다른 CSS에서 @import하여 사용하는 방식을 선택했다면 보다 효율적인 작업이 가능했을 것이라 생각했습니다.

➡️ Tailwind CSS 편의성

Tailwind CSS를 사용하면서 가장 인상 깊었던 점은 직관적이고 일관된 클래스 네이밍이었습니다.
정해진 규칙에 따라 요소별로 명확한 기준을 적용할 수 있어, 스타일 작업의 혼란을 줄이고 개발 시간을 단축하는 데 큰 도움이 되었습니다.

이 과정을 통해 Tailwind CSS의 높은 편의성을 체감할 수 있었으며,
변수명이나 커밋 메시지 등 모든 이름을 직관적이고 일관되게 작성하는 것의 중요성도 함께 느낄 수 있었습니다.
